# CountIn — Domain Model

## Purpose

This document is the authoritative reference for the core domain of CountIn.

It covers the five foundational entities that every other module depends on:

- **User** — the identity that owns or joins spaces
- **Space** — the top-level business entity (PG, Mess, Hostel, Co-Living)
- **SpaceMembership** — the access bridge between a User and a Space
- **Invitation** — the controlled onboarding gate
- **Member** — the operational profile of a person inside a Space

This document is the source of truth for:

- Database schema design
- JPA entity modelling
- Flyway migration scripts
- Service layer contracts
- API request / response design

---

## Entity Relationship Overview

The full system entity tree:

```
User ──────────────────────────────────────────────────────────────┐
 │                                                                  │ 1 : N (owner_id)
 │ 1 : N                                                            ▼
 ▼                                        ┌────────── Space ──────────────────────────────┐
SpaceMembership ──── N : 1 ──────────────►│                                               │
                                          │  1 : N           1 : N          1 : N         │
                                          │    │               │              │            │
                                          ▼    ▼               ▼              ▼            ▼
                                       Member  Invitation  Building         Menu       Notification
                                         │                    │               │
                                         │ 1 : N         1 : N │          1 : N │
                                         │                    ▼               ▼
                                      Payment              Floor           MenuOption
                                         │                    │               │
                                      Complaint          1 : N │           1 : N │
                                                              ▼               ▼
                                                            Room          MealResponse
                                                              │
                                                         1 : N │
                                                              ▼
                                                             Bed
```

**Entities covered in this document (foundational layer):**

- A **User** can have many **SpaceMemberships** across many Spaces.
- A **Space** has one owning User; every other entity hangs off a Space.
- An **Invitation** is issued by a Space and targets a mobile number.
- Accepting an Invitation creates a **SpaceMembership** and optionally a **Member** record.
- A **Member** captures the operational and financial profile of a person inside a Space.

**Downstream entities (to be documented separately):**

| Entity chain | Module |
|---|---|
| Space → Building → Floor → Room → Bed | Room Management |
| Space → Menu → MenuOption → MealResponse | Meal Management |
| Member → Payment | Payment Management |
| Member → Complaint | Complaint Management |
| Space → Notification | Notification |

---

## Global Rules

These rules apply to every entity in the system.

| Rule | Detail |
|------|--------|
| Primary key | UUID, generated by the database |
| Auditing | `createdAt` and `updatedAt` on every table, set automatically |
| No hard deletes | Business records are never physically deleted |
| Status transitions | Soft status changes replace deletion |
| Historical preservation | Joining date, exit date, and all status history must be retained |
| Database | PostgreSQL with `TIMESTAMPTZ` for all timestamps |

# User

## Purpose

A User represents a person who has registered on CountIn.

A User exists independently of any Space. The same person can own one Space, be a tenant in another, and be a meal customer in a third — all at the same time. There is no concept of a single-space user.

Authentication is mobile-number and OTP based. No password is stored.

---

## Fields

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | UUID | PK, not null | System-generated unique identifier |
| `mobileNumber` | String | Unique, not null | E.164 format, e.g. `+919876543210` |
| `fullName` | String | Not null | Display name |
| `profilePhotoUrl` | String | Nullable | URL to a stored profile image |
| `isActive` | Boolean | Not null, default `true` | Soft deactivation flag |
| `createdAt` | Timestamp | Not null, auto | Record creation time |
| `updatedAt` | Timestamp | Not null, auto | Last update time |

---

## Relationships

| Relationship | Cardinality | Description |
|--------------|-------------|-------------|
| User → SpaceMembership | One to Many | A user can be a member of many spaces |
| User → Space (as owner) | One to Many | A user can own many spaces via `owner_id` on Space |
| User → Invitation | One to Many | A user may have received multiple invitations across spaces |

---

## Status

A User does not have a lifecycle status enum. Active/inactive state is captured by the boolean `isActive`.

| Value | Meaning |
|-------|---------|
| `isActive = true` | Account is active and usable |
| `isActive = false` | Account is soft-disabled; user cannot log in or act |

---

## Business Rules

1. Mobile number must be globally unique across the entire system.
2. Mobile number is the only login identifier. No email, no password.
3. A User account must never be physically deleted.
4. Setting `isActive = false` is the only permitted deactivation mechanism.
5. A User can own multiple Spaces.
6. A User can hold memberships in multiple Spaces simultaneously.
7. A User's role can differ per Space — Owner in one, Tenant in another.
8. Profile photo is optional at registration but encouraged.
9. Full name is required and must not be blank.

---

# Space

## Purpose

A Space is the top-level business entity in CountIn. Every operational concept — members, rooms, menus, payments, complaints — exists within a Space.

A Space represents a real-world business such as a PG, a Mess, a Hostel, or a Co-Living facility.

---

## Fields

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | UUID | PK, not null | System-generated unique identifier |
| `ownerId` | UUID | FK → User, not null | The User who created and owns this Space |
| `name` | String | Not null | Display name, e.g. `PG-A`, `Office Mess` |
| `type` | Enum | Not null | Category of this space (see Space Types below) |
| `address` | String | Nullable | Physical address of the space |
| `contactNumber` | String | Nullable | Public contact number for the space |
| `isActive` | Boolean | Not null, default `true` | Whether this space is currently operating |
| `createdAt` | Timestamp | Not null, auto | Record creation time |
| `updatedAt` | Timestamp | Not null, auto | Last update time |

---

## Space Types

| Value | Description |
|-------|-------------|
| `PG` | Paying Guest accommodation |
| `MESS` | Meal-providing canteen or mess |
| `HOSTEL` | Hostel-style accommodation |
| `CO_LIVING` | Co-living or shared accommodation space |

---

## Relationships

| Relationship | Cardinality | Description |
|--------------|-------------|-------------|
| Space → User (owner) | Many to One | Each space has exactly one owning User |
| Space → SpaceMembership | One to Many | A space has many memberships |
| Space → Invitation | One to Many | A space has many invitations issued |
| Space → Member | One to Many | A space has many operational member profiles |
| Space → Building | One to Many | A space can contain multiple physical buildings |
| Space → Menu | One to Many | A space can have many daily menus |
| Space → Notification | One to Many | A space can send many notices and alerts |

---

## Status

A Space does not have a lifecycle status enum. Active/inactive state is captured by `isActive`.

| Value | Meaning |
|-------|---------|
| `isActive = true` | Space is open and operational |
| `isActive = false` | Space has been closed or archived |

---

## Business Rules

1. Every Space must have exactly one owner (`ownerId`).
2. The owner holds an implicit `OWNER` role via their SpaceMembership record.
3. One User can own multiple Spaces with no upper limit.
4. One Space can contain multiple Members.
5. A Space name must not be blank.
6. Space type must be one of the four permitted enum values.
7. A Space must never be physically deleted.
8. Setting `isActive = false` is the only way to retire a Space.
9. Contact number and address are optional at creation but can be updated later.

# SpaceMembership

## Purpose

SpaceMembership is the join entity that links a User to a Space. It carries the access role and the full lifecycle status of that relationship.

Every interaction a User has with a Space is governed by their SpaceMembership record. A User must have an ACTIVE membership with an appropriate role before they can perform any operation inside a Space.

---

## Fields

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | UUID | PK, not null | System-generated unique identifier |
| `userId` | UUID | FK → User, not null | The User this membership belongs to |
| `spaceId` | UUID | FK → Space, not null | The Space this membership is for |
| `role` | Enum | Not null | The access role of this user in this space |
| `status` | Enum | Not null | Current lifecycle status of this membership |
| `invitationId` | UUID | FK → Invitation, nullable | The invitation that created this membership |
| `joinedAt` | Timestamp | Nullable | When the membership became ACTIVE |
| `exitedAt` | Timestamp | Nullable | When the membership was VACATED or REMOVED |
| `createdAt` | Timestamp | Not null, auto | Record creation time |
| `updatedAt` | Timestamp | Not null, auto | Last update time |

---

## Membership Roles

| Role | Description |
|------|-------------|
| `OWNER` | Created the space; full administrative access |
| `MANAGER` | Delegated admin; can manage members and operations |
| `TENANT` | Resident of a PG, Hostel, or Co-Living space |
| `CUSTOMER` | Mess or meal subscriber |
| `STAFF` | Operational staff member of the space |

---

## Membership Status

| Status | Description |
|--------|-------------|
| `INVITATION_SENT` | Invitation has been issued; user has not yet accepted |
| `ACCEPTED` | User accepted the invitation; membership is being prepared |
| `ACTIVE` | Full access granted; user is an active member |
| `INACTIVE` | Temporarily suspended; no operational access |
| `REMOVED` | Owner or manager removed the user; no access |
| `VACATED` | User has checked out or left; historical record retained |

---

## Status Lifecycle

```
Invitation created
        │
        ▼
INVITATION_SENT
        │
        ├── User accepts ──────────────► ACCEPTED ──► ACTIVE
        │
        ├── Invitation expires ────────► (Invitation EXPIRED, no membership created)
        │
        └── User rejects ──────────────► (Invitation REJECTED, no membership created)

ACTIVE
  │
  ├── Owner removes ──────────────────► REMOVED
  ├── User vacates / checks out ──────► VACATED
  └── Owner suspends ────────────────► INACTIVE
              │
              └── Owner reactivates ─► ACTIVE
```

---

## Relationships

| Relationship | Cardinality | Description |
|--------------|-------------|-------------|
| SpaceMembership → User | Many to One | Each membership belongs to one User |
| SpaceMembership → Space | Many to One | Each membership belongs to one Space |
| SpaceMembership → Invitation | Many to One | Optionally linked to the Invitation that created it |

---

## Unique Constraint

The combination of `(userId, spaceId)` must be unique.

A User can hold only one active membership record per Space at a time. If a user leaves and rejoins, the existing record is updated rather than duplicated.

---

## Business Rules

1. A SpaceMembership record must never be physically deleted.
2. Historical memberships, including VACATED and REMOVED records, must be retained forever.
3. A user can hold memberships in multiple Spaces simultaneously.
4. Only users with `OWNER` or `MANAGER` role may manage other memberships.
5. An `OWNER` membership is created automatically when a Space is created.
6. The `(userId, spaceId)` pair must be unique across all records.
7. `joinedAt` must be set when status transitions to `ACTIVE`.
8. `exitedAt` must be set when status transitions to `VACATED` or `REMOVED`.
9. Status can only move forward along the defined lifecycle; no arbitrary jumping.
10. A User cannot access any Space resource without an `ACTIVE` membership.

# Invitation

## Purpose

An Invitation is the controlled entry point into a Space. Users cannot join a Space by themselves. An Owner or Manager must first issue an Invitation to a mobile number. The targeted person receives it, logs in, and accepts it — at which point a SpaceMembership is created.

This enforces owner-controlled onboarding throughout the platform.

---

## Fields

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | UUID | PK, not null | System-generated unique identifier |
| `spaceId` | UUID | FK → Space, not null | The Space this invitation is for |
| `invitedByUserId` | UUID | FK → User, not null | The User (OWNER or MANAGER) who sent the invitation |
| `mobileNumber` | String | Not null | The mobile number being invited |
| `role` | Enum | Not null | The role that will be assigned on acceptance |
| `status` | Enum | Not null, default `PENDING` | Current status of this invitation |
| `expiresAt` | Timestamp | Not null | When this invitation is no longer valid |
| `acceptedAt` | Timestamp | Nullable | When the invitation was accepted |
| `createdAt` | Timestamp | Not null, auto | Record creation time |
| `updatedAt` | Timestamp | Not null, auto | Last update time |

---

## Invitation Status

| Status | Description |
|--------|-------------|
| `PENDING` | Invitation has been sent and is awaiting response |
| `ACCEPTED` | Target user accepted; SpaceMembership has been created |
| `REJECTED` | Target user declined the invitation |
| `EXPIRED` | Invitation was not acted upon before `expiresAt` |

---

## Status Lifecycle

```
Owner/Manager creates invitation
        │
        ▼
     PENDING
        │
        ├── User accepts before expiry ──► ACCEPTED ──► SpaceMembership created
        ├── User rejects ────────────────► REJECTED
        └── expiresAt passes ────────────► EXPIRED
```

---

## Relationships

| Relationship | Cardinality | Description |
|--------------|-------------|-------------|
| Invitation → Space | Many to One | Each invitation belongs to one Space |
| Invitation → User (invitedBy) | Many to One | The sender of the invitation |
| Invitation → SpaceMembership | One to One (optional) | Created only when the invitation is accepted |

---

## Business Rules

1. Only a User with `OWNER` or `MANAGER` role in a Space may create an invitation for that Space.
2. The `role` assigned in the invitation becomes the role of the resulting SpaceMembership.
3. An invitation is tied to a mobile number, not a User ID — the recipient may not yet have an account.
4. If the recipient does not have an account, they must register before accepting.
5. An invitation must have an expiry time set at creation.
6. Once a status is `ACCEPTED`, `REJECTED`, or `EXPIRED`, it cannot be changed.
7. A new invitation can be sent to the same mobile number if a prior one has expired or was rejected.
8. Only one `PENDING` invitation per `(spaceId, mobileNumber)` combination is allowed at a time.
9. Accepting an invitation must atomically create the SpaceMembership and set `acceptedAt`.
10. Invitation records must never be physically deleted.

---

# Member

## Purpose

A Member captures the operational and financial profile of a person inside a specific Space. While SpaceMembership controls access, Member records contain the business details — rent, deposit, food preferences, occupancy dates.

A Member record is created when an Owner or Manager registers a tenant or customer with the operational details needed to manage them day-to-day.

---

## Fields

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| `id` | UUID | PK, not null | System-generated unique identifier |
| `spaceId` | UUID | FK → Space, not null | The Space this member belongs to |
| `userId` | UUID | FK → User, nullable | Linked User account (null if not yet registered) |
| `membershipId` | UUID | FK → SpaceMembership, nullable | Linked membership record |
| `name` | String | Not null | Display name (copied from User or entered manually) |
| `mobileNumber` | String | Not null | Contact number for this member |
| `joiningDate` | Date | Not null | The date the member moved in or subscribed |
| `exitDate` | Date | Nullable | The date the member moved out or cancelled |
| `monthlyRent` | Decimal | Nullable | Rent or subscription amount per month |
| `foodIncluded` | Boolean | Not null, default `false` | Whether food is part of the membership |
| `foodCharges` | Decimal | Nullable | Separate food charge if food is included |
| `depositAmount` | Decimal | Nullable | Security deposit collected |
| `status` | Enum | Not null | Operational status of this member |
| `notes` | String | Nullable | Free-text notes from the owner or manager |
| `createdAt` | Timestamp | Not null, auto | Record creation time |
| `updatedAt` | Timestamp | Not null, auto | Last update time |

---

## Member Status

| Status | Description |
|--------|-------------|
| `ACTIVE` | Currently residing or subscribed; all operations permitted |
| `NOTICE_PERIOD` | Has given or been given notice; departure is imminent |
| `VACATED` | Has moved out; historical record retained |
| `INACTIVE` | Temporarily inactive; may be reactivated |

---

## Status Lifecycle

```
Member added by Owner/Manager
        │
        ▼
      ACTIVE
        │
        ├── Notice issued ──────────────► NOTICE_PERIOD
        │         │
        │         └── Moves out ────────► VACATED
        │
        ├── Moves out directly ─────────► VACATED
        │
        └── Owner suspends ────────────► INACTIVE
                    │
                    └── Reactivated ───► ACTIVE
```

---

## Relationships

| Relationship | Cardinality | Description |
|--------------|-------------|-------------|
| Member → Space | Many to One | Each member belongs to one Space |
| Member → User | Many to One (optional) | Linked to a User account when they register |
| Member → SpaceMembership | One to One (optional) | Linked to the access record |
| Member → Payment | One to Many | A member can have many payment records |
| Member → Complaint | One to Many | A member can raise multiple complaints |

---

## Business Rules

1. A Member record must never be physically deleted.
2. `joiningDate` must be set at creation and must not be changed retrospectively.
3. `exitDate` must be set when the status transitions to `VACATED`.
4. `monthlyRent` should be set for PG and Hostel members; optional for Mess customers.
5. `depositAmount` should be recorded when collected; it is retained for historical reference.
6. `foodIncluded = true` enables food charge tracking via `foodCharges`.
7. A Member can exist without a linked User account (pre-registration workflow).
8. When the User registers and accepts their Invitation, the Member record is linked via `userId` and `membershipId`.
9. A Space Owner or Manager may add notes to a Member record at any time.
10. Only one `ACTIVE` or `NOTICE_PERIOD` Member record should exist per User per Space at a time.

## Global Rules (Consolidated)

All entities in CountIn must follow these rules without exception.

| Rule | Requirement |
|------|-------------|
| Primary key | UUID on every table |
| Audit fields | `createdAt` and `updatedAt` on every table, managed by JPA auditing |
| No hard deletes | Business data is never physically removed from the database |
| Soft status | Status field transitions replace deletion |
| Historical preservation | Joining date, exit date, and prior statuses must always be retained |
| Database | PostgreSQL; use `TIMESTAMPTZ` for all timestamp columns |
| ORM | JPA / Hibernate with `@EntityListeners(AuditingEntityListener.class)` |
| Migrations | All schema changes through Flyway; `ddl-auto: validate` in production |

Records that must never be deleted:

- Users
- Spaces
- SpaceMemberships
- Invitations
- Members
- Payments
- Complaints
- Meal Responses

Status changes must always be preferred over physical deletion.
